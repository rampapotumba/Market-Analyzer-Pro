## Приоритет 1 — Быстрый эффект (критические фильтры)

### REQ-01: Поднять порог composite score для генерации сигнала

**Текущее состояние:** порог ±10 из ±100. Пропускает слишком много слабых сигналов.

**Требование:**
- Глобальный порог входа: `|composite_score| >= 15` (было 10)
- Для крипто инструментов (`market == "crypto"`): `|composite_score| >= 20`
- Применяется одинаково в `SignalEngine` (live) и `BacktestEngine` (бэктест)
- Константы выносятся в конфигурацию (не хардкодить)

**Ожидаемый эффект:** -25..30% количества сделок, +3..5% win rate за счёт отсечения слабых входов.

---

### REQ-02: Запрет торговли в режиме RANGING

**Текущее состояние:** в `RANGING` режиме система генерирует трендовые сигналы с min_rr=1.0, что математически эквивалентно breakeven.

**Требование:**
- Если `regime == "RANGING"` → сигнал не генерируется (return None)
- Применяется в `SignalEngine.generate_signal()` после определения режима
- Применяется в `BacktestEngine._generate_signal()` аналогично
- Логировать пропуск: `[SignalEngine] Skipping: RANGING regime for {symbol}`
- Настраиваемый список блокируемых режимов (чтобы можно было добавить другие)

**Обоснование:** трендовые сигналы в боковике имеют случайное направление по природе. Это один из самых распространённых источников ложных входов.

### REQ-03: Фильтр дневного тренда (D1 MA200 alignment)

**Текущее состояние:** система может открывать LONG позицию пока D1 находится в даунтренде, и наоборот. Контртрендовые входы на H1 имеют статистически худшую вероятность успеха.

**Требование:**
- Перед генерацией сигнала на H1/H4 проверять дневной тренд по MA200:
  - `LONG` сигнал: допустим только если `close(D1) > MA200(D1)`
  - `SHORT` сигнал: допустим только если `close(D1) < MA200(D1)`
- Для D1 сигналов: проверять W1 MA50 (недельный тренд)
- Для M1/M5/M15 сигналов: фильтр не применяется (скальпинг — другая логика)
- Данные D1 берутся из `price_data` таблицы (тайм-фрейм `D1`)
- Если D1 данных нет — пропустить фильтр с предупреждением (не блокировать)
- В `BacktestEngine`: реализовать тот же фильтр используя исторические D1 candles без lookahead

**Ожидаемый эффект:** устранение контртрендовых входов, улучшение win rate на +3..6%.

### REQ-04: Исключить BTC/USDT из стандартной конфигурации / увеличить параметры

**Текущее состояние:** BTC/USDT — самый убыточный символ (-$269 за 154 сделки, WR 40.9%). Причина: стандартные ATR-мультипликаторы (2.5×) слишком малы для волатильности BTC.

**Требование (вариант A — рекомендуемый):**
- Добавить в конфигурацию `INSTRUMENT_OVERRIDES` словарь с per-symbol параметрами:
  ```python
  INSTRUMENT_OVERRIDES = {
      "BTC/USDT": {
          "sl_atr_multiplier": 3.5,   # было 2.5 (VOLATILE режим)
          "min_composite_score": 20,  # был глобальный 15
          "allowed_regimes": ["STRONG_TREND_BULL", "STRONG_TREND_BEAR"],
      },
      "ETH/USDT": {
          "sl_atr_multiplier": 3.5,
          "min_composite_score": 20,
      },
  }
  ```
- `RiskManagerV2.calculate_levels_for_regime()` принимает override параметры
- `BacktestEngine._generate_signal()` использует те же overrides
- Альтернативный вариант B: временно исключить крипто из списка инструментов симулятора через флаг `is_backtestable` в таблице `instruments`

## Приоритет 2 — Структурные улучшения сигналов

### REQ-06: Volume confirmation filter

**Текущее состояние:** сигналы генерируются без учёта объёма. Паттерн без подтверждения объёмом статистически является ложным сигналом.

**Требование:**
- Вычислять скользящее среднее объёма за 20 периодов: `vol_ma20 = mean(volume[-20:])`
- Условие входа: `volume[-1] >= 1.2 × vol_ma20` (объём текущей свечи >= 120% от среднего)
- Если объём ниже — не генерировать сигнал (не снижать score, а полностью блокировать)
- Исключение: если объём не передаётся брокером (все значения 0) — фильтр не применяется
- Реализовать как отдельный метод `_check_volume_confirmation(df) -> bool` в `SignalEngine`
- Применить в `BacktestEngine._generate_signal()` аналогично

---

### REQ-07: Импульсный фильтр (Momentum confirmation)

**Текущее состояние:** система может генерировать LONG сигнал когда MACD падает, или SHORT когда RSI растёт — противоречие направлений.

**Требование:**
- Перед генерацией сигнала проверять согласованность импульса:
  - `LONG`: RSI(14) > 50 И MACD line > Signal line
  - `SHORT`: RSI(14) < 50 И MACD line < Signal line
- Если импульс не подтверждает направление — не генерировать сигнал
- Реализовать как отдельный метод `_check_momentum_alignment(ta_indicators, direction) -> bool`
- Значения RSI и MACD берутся из `ta_engine.calculate_all_indicators()`
- Если TAEngine не возвращает эти значения — фильтр пропускается с предупреждением (не блокирует)

---

### REQ-08: Фильтр по Score bucket — минимальный уровень силы сигнала

**Текущее состояние:** система принимает сигналы начиная с `BUY` (score ≥ 10), но `weak_buy` (7–10) и пограничные значения дают плохое качество.

**Требование:**
- В дополнение к REQ-01, ввести минимальную силу сигнала для открытия позиции:
  - Допустимые значения: `BUY`, `STRONG_BUY`, `SELL`, `STRONG_SELL`
  - Запрещены: `HOLD`, `WEAK_BUY`, `WEAK_SELL`
- Привязать к конфиг-параметру `MIN_SIGNAL_STRENGTH` (default: `BUY`)
- Логировать отфильтрованные слабые сигналы на уровне DEBUG

---

### REQ-09: Weekday filter — запрет торговли в понедельник (открытие) и пятницу (закрытие)

**Текущее состояние:** нет фильтра по дням недели. Понедельники характеризуются гэпами, пятницы — закрытием институциональных позиций перед выходными.

**Требование:**
- Понедельник 00:00–10:00 UTC: не генерировать новые сигналы (гэп-период открытия рынка)
- Пятница 18:00–23:59 UTC: не генерировать новые сигналы (форекс закрывается в пятницу)
- Для крипто: фильтр понедельника не применяется (24/7 рынок)
- Для акций/ETF: фильтр применяется аналогично форекс

### REQ-10: Фильтр экономического календаря в бэктесте

**Текущее состояние:** в live-симуляторе есть блок на HIGH-impact события, в бэктесте — нет. Это создаёт расхождение результатов.

**Требование:**
- Сохранять исторические economic events в таблицу `economic_events` при наличии данных (или добавить поле `historical: bool`)
- В `BacktestEngine._simulate_symbol()`: перед генерацией сигнала проверять, есть ли HIGH-impact event в течение ±2 часов от timestamps свечи
- Если событие есть — пропустить свечу (аналогично `calendar_block` в live)
- Если исторических событий нет в БД — фильтр пропускается (не блокирует работу)

## Приоритет 3 — Управление позициями

### REQ-11: Breakeven на 50% пути, а не в entry

**Текущее состояние:** после partial close (50% at TP1) SL переносится точно в entry. Рынок часто откатывает от TP1 до entry и выбивает оставшуюся позицию до достижения TP2.

**Требование:**
- После partial close переносить SL не в entry, а на уровень `entry + 0.5 × (TP1 - entry)` для LONG / `entry - 0.5 × (entry - TP1)` для SHORT
- Это обеспечивает буферную зону и защищает от нормального рыночного отката
- Параметр буфера: `BREAKEVEN_BUFFER_RATIO = 0.5` (50% пути к TP1), конфигурируемый
- Применяется в `SignalTracker._partial_close()`
- Применяется в `TradeLifecycleManager` для trailing/breakeven логики

---

### REQ-12: Time-based exit — закрытие застрявших позиций

**Текущее состояние:** позиция может висеть 48+ часов без движения (avg winner = 35h на H1). Это "dead money", который тянет drawdown и capital utilization.

**Требование:**
- Добавить проверку: если позиция открыта более `N` свечей (по таймфрейму) и не достигла безубытка — закрыть по текущей цене
- Порог по таймфрейму:
  - H1: 48 свечей (48 часов)
  - H4: 20 свечей (~3.3 дня)
  - D1: 10 свечей (2 недели)
- `exit_reason = "time_exit"`
- Параметр `TIME_EXIT_CANDLES` — конфигурируемый, per-timeframe
- Проверять в `SignalTracker.check_signal()` после MAE early exit проверки

---

### REQ-13: S/R snapping для SL в бэктесте

**Текущее состояние:** в live-симуляторе `RiskManagerV2._align_sl_long/short()` выравнивает SL по ближайшему S/R уровню. В бэктесте SL ставится чисто по ATR без структурных уровней.

**Требование:**
- В `BacktestEngine._recalc_sl_tp()` передавать S/R уровни из TAEngine при их наличии
- TAEngine должен возвращать `support_levels` и `resistance_levels` через `calculate_all_indicators()`
- Использовать `RiskManagerV2.calculate_levels_for_regime()` с уровнями (сейчас вызывается без них)
- Это сделает бэктест более достоверным относительно live поведения

### REQ-14: Обновление swap-ставок

**Текущее состояние:** `SWAP_DAILY_PIPS` в `signal_tracker.py` содержат значения, которые не отражают текущую денежно-кредитную политику (данные 2023 года).

**Требование:**
- Создать механизм загрузки актуальных swap-ставок из внешнего источника (Pepperstone API, OANDA API или статический файл с обновлением)
- Или вынести `SWAP_DAILY_PIPS` в конфигурационный файл `config/swap_rates.json` для ручного обновления
- Добавить дату последнего обновления и предупреждение если данным > 90 дней
- Обязательно добавить ставки для AUDUSD, которого нет в текущей таблице (используется в бэктесте)

## Приоритет 4 — Новые источники данных

### REQ-15: DXY (Dollar Index) как real-time фильтр

**Текущее состояние:** `CorrelationEngine` частично учитывает DXY через macro_records, но не в real-time.

**Требование:**
- Собирать DXY (DX-Y.NYB) через yfinance в `realtime_collector` с интервалом 1 минута
- В `SignalEngine`: если DXY растёт (RSI(DXY) > 55) — блокировать LONG для USD-denominated пар (EURUSD, GBPUSD, AUDUSD, NZDUSD) и разрешать LONG для USD-base пар (USDJPY, USDCAD, USDCHF)
- Обратная логика при падении DXY
- Реализовать как отдельный метод `_check_dxy_alignment(direction, symbol) -> bool` в `CorrelationEngine`

### REQ-16: Fear & Greed Index для крипто

**Текущее состояние:** нет использования Fear & Greed Index, который является ключевым сентимент-индикатором для крипторынка.

**Требование:**
- Собирать Fear & Greed Index из API `alternative.me/fng/` (бесплатно, без ключа)
- Хранить последние 30 значений в БД (таблица `macro_data` или отдельная `fear_greed_index`)
- Правила использования:
  - `value <= 20` (Extreme Fear) → усиливать LONG сигналы для BTC/ETH (+5 к composite)
  - `value >= 80` (Extreme Greed) → усиливать SHORT сигналы для BTC/ETH (+5 к composite)
  - Нейтральная зона (21–79) → не влиять
- Собирать раз в час через APScheduler

### REQ-17: Funding Rate Extreme Filter для крипто

**Текущее состояние:** funding rate собирается, но не используется как фильтр сигналов (только в swap расчёте).

**Требование:**
- Если funding rate > +0.1% (8h) — рынок перегрет лонгами → LONG сигналы блокируются или штрафуются (-10 к composite)
- Если funding rate < -0.1% (8h) — рынок перегрет шортами → SHORT сигналы блокируются или штрафуются (-10 к composite)
- Применяется только для крипто инструментов
- Реализовать в `CryptoFAEngine.analyze()` как дополнительный компонент

### REQ-18: COT Data (Commitment of Traders) для форекс

**Текущее состояние:** не используется. COT данные показывают позиционирование крупных участников рынка.

**Требование:**
- Собирать COT данные из CFTC API (публичный, еженедельно по пятницам)
  - URL: `https://www.cftc.gov/dea/newcot/deacot.htm`
- Хранить в таблице `macro_data` с `indicator = "COT_{symbol}"`, `country = "US"`
- Поля: net positions (commercials, non-commercials), change_week
- Интерпретация:
  - Non-commercials (speculators) net long и увеличивают → подтверждение LONG
  - Non-commercials net short и увеличивают → подтверждение SHORT
- Использовать в `FAEngine.calculate_fa_score()` как дополнительный компонент (+/- 5 к FA score)
- Обновлять раз в неделю (пятница, после публикации CFTC)

## Приоритет 5 — Достоверность бэктеста

### REQ-19: Унификация логики бэктеста и live симулятора

**Текущее состояние:** несколько фильтров работают в live, но отсутствуют в бэктесте, что делает результаты несопоставимыми.

**Требование — все следующие фильтры должны работать одинаково в live и backtest:**
- Session filter (Asian session block) ✅ уже реализован
- Cooldown per timeframe ✅ уже реализован
- Composite score threshold (REQ-01) — применить в бэктесте
- RANGING regime block (REQ-02) — применить в бэктесте
- D1 MA200 trend filter (REQ-03) — применить в бэктесте
- Volume confirmation (REQ-06) — применить в бэктесте
- Momentum filter (REQ-07) — применить в бэктесте
- Weekday filter (REQ-09) — применить в бэктесте
- Economic calendar filter (REQ-10) — применить в бэктесте

**Принцип:** если фильтр существует в live, он должен существовать в бэктесте. Бэктест без всех фильтров — это не бэктест реальной стратегии.

### REQ-21: Хранение и отображение дополнительных метрик бэктеста

**Текущее состояние:** summary содержит базовые метрики. Не хватает детализации для принятия решений.

**Требование — добавить в summary бэктеста:**
